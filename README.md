# girin-codex

**내가그린기린기록** 프로젝트의 단일 진실 공급원(SSOT)이자 하네스 규칙 저장소.

백엔드 2명 · 프론트엔드 1명이 바이브 코딩으로 협업할 때,
세 사람(과 각자의 에이전트)이 **같은 계약·같은 규칙·같은 강제 장치**를 바라보게 만드는 곳이다.
애플리케이션 코드는 여기 없다. 여기 있는 건 "모두가 따라야 하는 것"뿐이다.

## 무엇이 여기 사는가

| 경로 | 역할 |
| --- | --- |
| `api/openapi.yaml` | **계약(Contract).** 모든 엔드포인트의 단일 진실. BE는 이대로 구현하고, FE는 이대로 호출한다. |
| `conventions/` | 코딩·API·Git·용어 규칙. 에이전트가 표류하지 않게 잡아주는 가이드. |
| `domain/data-model.md` | 엔티티 정의(User, Persona, Diary, Retrospective, …). |
| `harness/` | BE/FE 레포에 복사해 넣는 **강제 장치(hook)** 템플릿과 설치 방법. |
| `CLAUDE.md` | 이 레포에서 작업하는 에이전트의 컨텍스트. |

## 왜 별도 레포인가

규칙을 BE 레포나 FE 레포 한쪽에 두면, 다른 쪽은 그 규칙을 "남의 것"으로 본다.
계약과 규칙을 중립 지대에 두고 양쪽이 **참조(submodule / 패키지 / 복사)** 하게 하면,
- 명세가 한 곳에서만 바뀌고,
- 타입이 한 소스에서 생성되며,
- hook 설정이 세 사람 모두 동일하게 적용된다.

## 핵심 원칙 (단 하나만 기억한다면)

> **`api/openapi.yaml`이 옳다. 코드가 명세와 다르면, 틀린 건 코드다.**

명세는 사람이 합의해서 바꾸고(PR + 리뷰), 코드는 명세를 따른다.
이 한 줄을 강제하는 게 `harness/`의 hook들이다.

## 빠른 시작

1. 이 레포를 만든다(아래 "초기화" 참고).
2. 실제 API 명세를 받으면 `api/openapi.yaml`에 채운다.
3. `conventions/`의 `[확정 필요]` 표시를 팀이 합의해서 채운다.
4. BE/FE 레포에서 `harness/SETUP.md`대로 hook을 설치한다.
5. 끝. 이제 세 사람의 에이전트가 같은 규칙 위에서 코딩한다.

### 초기화

```bash
git init
git add .
git commit -m "chore: girin-codex 초기 구조 추가"
# 원격 레포 연결 후
git push -u origin main
```

## 변경 규칙

- 이 레포의 변경은 **항상 PR**로 한다. 직접 push 금지(특히 `api/`, `conventions/`).
- 명세(`api/`)나 용어(`glossary.md`)를 바꾸면 BE/FE 양쪽에 영향이 가므로, **PR 설명에 영향 범위를 적는다.**
- merge되면 BE/FE는 타입 재생성 / submodule 업데이트를 돌린다.
