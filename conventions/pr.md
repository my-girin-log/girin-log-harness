# PR 작업 단위 컨벤션

내가그린기린기록의 모든 레포(`girin-log-back` · `girin-log-front` · `girin-log-harness`)가 공유하는 PR 분할 기준이다.

목적은 PR을 무조건 작게 만드는 것이 아니라, 사람과 Codex가 하나의 목적을 독립적으로 이해하고 검증할 수 있게 만드는 것이다.

> 좋은 PR은 작은 PR이 아니라, 하나의 목적을 독립적으로 이해하고 검증할 수 있는 PR이다.

## 1. 기본 원칙

- 하나의 PR은 하나의 변경 목적을 가진다.
- 하나의 PR은 하나의 주요 검증 질문에 답할 수 있어야 한다.
- PR은 그 자체로 빌드, 테스트, 리뷰가 가능한 상태여야 한다.
- PR을 나누는 이유와 나누지 않는 이유는 모두 리뷰 이해도를 기준으로 판단한다.

좋은 검증 질문 예시:

- Memo 생성이 요구사항대로 동작하는가?
- GitHub OAuth 콜백에서 신규/기존 사용자를 올바르게 처리하는가?
- DailyChatSession 종료 정책이 도메인 규칙과 일치하는가?

## 2. 변경 목적이 두 개 이상인 경우

PR의 변경 내용을 한 문장으로 설명했을 때 `그리고`가 필요하다면 분리를 검토한다.

예시:

- 회원 가입 기능을 구현하고 예외 처리 구조를 변경한다.
- 메모 생성 API를 추가하고 기존 조회 로직을 리팩터링한다.
- 결제 기능을 구현하고 공통 응답 형식을 수정한다.

가능하면 다음과 같이 분리한다.

1. 공통 예외 처리 구조 변경
2. 회원 가입 기능 구현

단, 두 변경이 분리될 수 없을 정도로 강하게 연결되어 있다면 하나의 PR에 포함할 수 있으며, PR 본문에 함께 변경해야 하는 이유를 작성한다.

## 3. 기능 구현과 관련 없는 리팩터링이 섞인 경우

기능 구현 중 발견한 개선 사항이 현재 기능을 완성하는 데 필수적이지 않다면 별도의 PR로 분리한다.

분리 대상 예시:

- 패키지 구조 변경
- 클래스명 및 메서드명 일괄 수정
- 공통 응답 객체 리팩터링
- 사용하지 않는 코드 제거
- 코드 포맷 전체 변경
- 기존 테스트 구조 변경

현재 기능을 구현하기 위해 반드시 필요한 최소한의 리팩터링은 같은 PR에 포함할 수 있다.

리팩터링 PR은 다음을 지킨다.

- 기능 동작 변경 없음
- API 응답 변경 없음
- DB 스키마 변경 없음
- 테스트 기대값 변경 최소화
- 변경 이유와 영향 범위 명시

## 4. 독립적으로 머지할 수 있는 변경인 경우

다른 변경 없이도 실행, 테스트, 리뷰가 가능한 작업이라면 별도의 PR로 분리한다.

예시:

- DB 마이그레이션 추가
- 공통 예외 코드 추가
- 외부 API 클라이언트 생성
- 도메인 객체 추가
- 조회 성능 개선
- 로그 및 모니터링 추가

단순히 분리가 가능하다는 이유만으로 지나치게 작은 PR을 만들지는 않는다. 분리했을 때 리뷰와 작업 흐름이 더 명확해지는지를 함께 판단한다.

## 5. 선행 작업과 후속 작업이 구분되는 경우

작업 사이에 명확한 의존 순서가 있다면 선행 작업을 먼저 분리한다.

예시:

1. 공통 응답 형식 추가
2. 해당 응답 형식을 사용하는 API 구현

1. 테이블 및 엔티티 추가
2. 저장 기능 구현
3. 조회 API 구현

1. 외부 API 클라이언트 구현
2. 외부 API를 사용하는 비즈니스 로직 구현
3. 타임아웃 및 재시도 정책 추가

선행 PR은 그 자체로 프로젝트를 깨뜨리지 않는 상태여야 한다.

## 6. 리뷰어가 한 번에 이해하기 어려운 경우

다음과 같은 상황이면 PR 분리를 검토한다.

- 여러 도메인이 동시에 변경된다.
- 요청부터 응답까지의 흐름이 여러 개 포함된다.
- 중요한 설계 판단이 여러 개 들어 있다.
- 트랜잭션, 동시성, 예외 처리 변경이 한 PR에 함께 포함된다.
- 변경된 코드의 목적을 설명하기 위해 긴 사전 설명이 필요하다.
- 리뷰어가 변경 전후 흐름을 한 번에 추적하기 어렵다.

PR의 크기보다 리뷰어가 이해해야 하는 맥락의 개수를 기준으로 판단한다.

## 7. 테스트 관점이 서로 다른 경우

서로 다른 테스트 목적이 포함되어 있다면 PR 분리를 검토한다.

예시:

- 도메인 규칙 단위 테스트 추가
- Controller부터 DB까지의 통합 테스트 추가
- 외부 API 연동 테스트 추가
- 성능 테스트 추가
- 동시성 테스트 추가

기능 구현에 반드시 필요한 기본 테스트는 같은 PR에 포함한다. 성능, 부하, 동시성처럼 별도의 환경이나 설명이 필요한 테스트는 후속 PR로 분리할 수 있다.

## 8. 병렬 작업 시 충돌 가능성이 높은 경우

두 개발자가 동시에 같은 공통 파일을 수정해야 한다면 공통 변경을 먼저 별도 PR로 분리한다.

충돌 가능성이 높은 파일 예시:

- `SecurityConfig`
- `GlobalExceptionHandler`
- 공통 `ErrorCode`
- 공통 응답 객체
- DB 마이그레이션 파일
- API 문서
- 공통 설정 파일

공통 변경을 먼저 머지한 뒤 각 기능 브랜치에 반영한다.

## 9. 계약 변경과 구현 변경이 함께 있는 경우

OpenAPI, 요구사항, 도메인 규칙이 바뀌는 작업은 구현 PR과 분리한다.

권장 순서:

1. 요구사항/계약 변경 PR
2. 서버 구현 PR
3. 프론트 연동 PR

`girin-log-harness`에서 계약 변경 PR이 머지된 뒤, 제품 레포는 submodule 또는 참조 파일을 갱신하고 구현 PR을 올린다.

단, 문서 오타 수정이나 구현 영향이 없는 설명 보강은 같은 PR에 포함할 수 있다.

## 10. 변경 파일이 과도하게 넓게 퍼지는 경우

다음 기준은 절대적인 제한이 아니라 분할을 검토하기 위한 신호로 사용한다.

- 핵심 변경 파일이 10개 이상이다.
- 여러 패키지 또는 여러 도메인을 동시에 수정한다.
- 테스트를 제외한 변경 코드가 약 300~500줄을 넘는다.
- 리뷰에 약 20~30분 이상이 필요할 것으로 예상된다.
- 하나의 PR에서 2개 이상의 API 흐름을 새로 구현한다.

여기서 핵심 변경 파일은 실제 동작을 바꾸는 프로덕션 코드, API 계약, DB 스키마, 도메인 정책 파일을 의미한다. 테스트, 문서, fixture, 단순 포맷 변경은 별도로 판단한다.

파일 수나 코드 줄 수가 많더라도 하나의 변경 목적을 가진 반복적인 수정이라면 하나의 PR로 유지할 수 있다.

## 11. 분리하면 안 되는 경우

다음 경우에는 억지로 PR을 나누지 않는다.

- 선행 PR만 머지하면 빌드가 깨진다.
- 선행 PR만 보면 변경 의도를 이해할 수 없다.
- 후속 PR 없이는 테스트할 수 없다.
- 단순히 파일 수를 줄이기 위해 응집된 하나의 기능 흐름을 쪼개는 경우.
- 사용되지 않는 추상화, DTO, 인터페이스만 먼저 추가하는 경우.

작게 나눈 PR도 그 자체로 의미가 있어야 한다.

## 12. PR 본문 권장 템플릿

```markdown
## 변경 목적

## 변경 범위

## 리뷰어가 집중해서 볼 부분

## 제외한 것

## 테스트

## 관련 문서/이슈
```

`제외한 것`에는 이번 PR에서 의도적으로 하지 않은 일을 적는다.

예시:

```markdown
## 제외한 것

- MemoSummary 수정 API는 MVP 범위가 아니므로 추가하지 않음
- Diary 생성 배치는 후속 PR에서 처리
```

## 13. PR 올리기 전 체크리스트

- [ ] PR 제목을 한 문장으로 설명했을 때 `그리고`가 필요하지 않다.
- [ ] 기능 구현과 필수적이지 않은 리팩터링이 섞이지 않았다.
- [ ] 계약 변경과 구현 변경이 한 PR에 섞이지 않았다.
- [ ] 리뷰어가 집중해서 볼 부분이 PR 본문에 적혀 있다.
- [ ] 이번 PR에서 제외한 작업이 있다면 명시했다.
- [ ] 빌드와 테스트가 독립적으로 실행 가능한 상태다.
