# 테스트 규칙

이 문서는 **무엇을, 어디까지, 어떻게 테스트할지**에 대한 약속이다.
`conventions/coding.md`(설계)·`conventions/api.md`(명세)·`conventions/persistence.md`(저장 경계)와 함께 읽는다.
우테코에서 정리한 테스트 원칙을 이 서비스(회고 도구) 도메인에 맞춰 옮긴 것이다.
"지향"은 이 문서가, "강제"는 `harness/`의 hook이 맡는다.

## 0. 테스트의 목적 — 회귀 보호

테스트가 보호하는 것은 **"지금 옳다"가 아니라 "바꿔도 유지된다"** 는 회귀 안전성이다.

- 코드를 변경·리팩터링한다면, 기존 동작이 깨지지 않았음을 보장하는 테스트가 **먼저** 있어야 한다.
- 그래서 "무엇을 테스트하지 않을지"가 "무엇을 테스트할지"만큼 중요하다(4절).
- 깨졌을 때 **무엇이 왜 깨졌는지 즉시 알 수 있는** 테스트가 좋은 테스트다.

## 1. 테스트 단위 — 행위(Behavior)

메서드 한 개당이 아니라 **도메인의 행위 하나당** 테스트 하나를 둔다.
"이 객체가 어떤 결정을 내리는가"를 기준으로 자른다.

- `public` 메서드는 테스트한다(단, getter/단순 위임 제외). `private` 메서드는 따로 테스트하지 않는다 — 객체 자율성을 위해 내부 구현을 노출하지 않는다.
- **테스트 형식: given–when–then.** 세 단계가 눈에 보이게 나눈다.
- **`@DisplayName`은 한국어로 "행위·기대 결과"를 적는다.** 메서드명이 아니라 규칙을 쓴다.
  - 예: `@DisplayName("역질문이 10회에 도달하면 세션은 MAX_FOLLOWUP으로 종료된다")`
- **하나의 테스트는 하나의 개념만** 검증한다. 단언이 여러 규칙에 걸치면 쪼갠다.
- **단언은 AssertJ(`assertThat`)로 통일.** JUnit `assertEquals`를 섞지 않는다. 예외는 `assertThatThrownBy`.

### TDD: Red → Green → Refactor

도메인 로직과 핵심 정책은 가능하면 테스트를 먼저 쓴다.

1. **Red** — 구현 없이, 요구사항(행위)을 검증하는 실패하는 테스트를 먼저 쓴다.
2. **Green** — 통과시키기 위해 가장 단순하게 구현한다.
3. **Refactor** — 테스트가 통과하는 상태를 유지하며 중복 제거·가독성 개선.

## 2. 검증 수준 — 단위 / 통합 / E2E

| 수준 | 대상 | 도구 |
| --- | --- | --- |
| 단위 | JVM 안에서 닫히는 순수 로직(도메인 규칙·계산·상태 전이) | JUnit 5 + AssertJ (`new` + 외부 의존 제거) |
| 통합 | 외부 의존(Spring·DB·HTTP)이 동작의 일부인 경우 | `@SpringBootTest` + Testcontainers PostgreSQL |
| E2E | 사용자 시나리오 전체 보호 | 통합 환경 + 실제 요청 흐름 |

**수준을 정하는 순서:**

1. 외부 의존 없이 JVM 안에서 표현 가능한가? → **단위**
2. 외부 의존과의 결합 자체가 동작인가? → **통합**
3. 사용자 시나리오 전체 보호가 목적인가? → **E2E**

피라미드를 지킨다: **단위 다수 → 통합 소수**. 모든 것을 무거운 `@SpringBootTest`로 검증하지 않는다.

## 3. 실제 DB / Fake / Mock — 무엇을 언제 쓰나

협력자를 무엇으로 대체할지는 **"검증 대상의 본질이 상태냐 상호작용이냐"** 로 가른다.

- **실제 DB (Testcontainers PostgreSQL)** — 보호 대상이 DB의 상태 그 자체(SQL·매핑·제약조건·실제 쓰기/읽기)일 때. SQL·매핑은 가짜로 재현할 수 없다. 운영과 같은 방언으로 검증한다(H2는 제약이 있을 때만 대안).
- **Fake (인메모리 Repository)** — DB가 검증 대상이 아니라 협력자이고, 그 위 비즈니스 로직만 격리해서 보고 싶을 때. Repository의 본질은 "호출됐다"가 아니라 **"저장·조회의 결과 상태"** 다. `save` 한 것을 `findById`로 다시 꺼내는 흐름은 내부 상태를 가진 Fake라야 자연스럽게 재현된다.
- **Mock** — 협력자가 **보관할 상태가 없는 통신**일 때. 실록이 LLM 호출(`silok/`)·외부 HTTP가 여기 해당한다. `request → response` 한 번의 통신이 전부이므로 "어떤 호출이 어떤 인자로 일어났는가"라는 상호작용을 검증한다.

**금지 — 영속성 계층(Repository/JDBC)을 Mock으로 단위 테스트하지 않는다.**
정작 검증해야 할 SQL이 mock 뒤로 숨고, 상태가 없는 mock은 `save → find` 흐름 자체를 표현하지 못한다. 결국 내가 기대한 동작을 내가 다시 적어 넣는 자기충족적 테스트가 된다.

> 주의: 트랜잭션·롤백처럼 **DB의 동작 자체가 검증 대상**이 되는 순간에는 DB를 Fake로 빼면 검증 가치가 사라진다. 이때는 실제 DB로 본다(`conventions/persistence.md`).

## 4. 테스트하지 않는 것 (금지)

통과해도 새로 보증되는 게 없는 테스트는 잡음이다. 다음은 테스트하지 않는다.

- **자체 로직이 없는 것** — getter/setter·생성자·`equals`/`hashCode`·단순 위임. 깨질 분기가 없고, 깨지면 컴파일러·IDE가 잡는다.
- **우리 소유가 아닌 것** — 프레임워크·라이브러리·외부 시스템의 동작. `List.add()`나 JPA가 잘 도는지 검증하지 않는다. 우리가 볼 건 "그걸 올바르게 사용했는가"뿐이다.
- **이미 다른 테스트가 보증하는 것** — 하위에서 검증된 로직을 상위 메서드에서 재검증하지 않는다. 단, 그 메서드를 **활용해 새 로직**(결과 판별·점수 계산 등)이 생기면 그 부분은 다시 검증한다.
- **입출력(I/O) 자체** — 순수 계산 결과만 본다.
- **책임 범위 밖 케이스** — 그 기능이 책임지지 않는 입력을 굳이 만들어 검증하지 않는다.

## 5. 테스트가 어려우면 = 설계 신호

**테스트하기 어려운 코드는 설계가 잘못됐다는 신호다.** 테스트를 억지로 끼워 맞추지 말고 대상을 리팩터링한다.

- **무작위·현재 시각에 의존하지 않는다.** 06:00 KST 일자 계산은 `Clock`(또는 KST 유틸)을 주입해 시간을 고정한다. `LocalDate.now()` 직접 호출 금지. 무작위가 필요한 곳도 값으로 주입받는다.
- **실록이 LLM 응답은 주입 가능한 경계 뒤에 둔다**(`silok/`). 도메인 로직 테스트가 실제 LLM 호출에 의존하지 않게 한다.
- 메서드가 2개 이상의 도메인 로직을 품고 있어 테스트가 엉키면, 각 로직을 별도 메서드로 분리하고 각각 테스트한다.
- 테스트 준비(given)가 길어지면 헬퍼/픽스처로 정리한다.

## 6. 이 서비스에서 반드시 테스트할 정책

기획 리뷰에서 합의된 핵심 규칙이다. "있으면 좋은" 테스트가 아니라 **필수**다(대부분 단위 테스트로 가능).

- **일자 경계 06:00 KST** — 05:59에 남긴 메모는 전날, 06:00에 남긴 메모는 당일로 묶인다. 경계 직전·직후를 모두 본다.
- **역질문 최대 10회** — 9/10/11회 경계. 10회 도달 시 더 질문하지 않고 `endedReason=MAX_FOLLOWUP`으로 종료된다. **서버 정책 레벨**에서 막히는지 확인(프롬프트만 신뢰 금지).
- **세션 종료 사유** — `USER_ENDED`/`MAX_FOLLOWUP`/`AI_DECIDED`가 올바른 상황에서 기록된다.
- **Diary는 하루 1개** — 하루에 여러 Memo·여러 DailyChatSession이 있어도 06:00 KST 정리 결과 Diary는 날짜당 정확히 하나다.
- **Memo 상태 전이** — `DRAFT → SUMMARIZED`(요약 시), 06:00 KST 이후 `ARCHIVED`. 정의되지 않은 전이는 거부된다.
- **Persona 부분 입력 생성** — 블로그/원문/설문 중 일부만 있어도 Persona가 생성된다.
- **MemoSummary 불변** — MVP에서 수정/삭제 경로가 없다(엔드포인트 부재 또는 거부).
- **MemoSummary 재선택 금지** — 이미 DailyChatSession에 사용된 MemoSummary는 `chatAvailable=false`로 조회되고, 다시 세션 시작에 사용하면 `422`로 거부된다.
- **사용자 종료는 완전 종료** — 사용자가 중간에 끝낸 세션은 `status=ENDED`, `endedReason=USER_ENDED`가 되며 이후 답변 제출로 이어서 대화할 수 없다.
- **작업 공간 초기화는 보관** — 06:00 초기화 후에도 이전 데이터가 삭제되지 않고 조회·Diary 생성에 쓰인다.

## 7. 명세 준수 테스트

각 도메인은 "응답이 `api/openapi.yaml`과 일치하는가"를 검증하는 테스트를 가진다.

- 성공 응답의 필드명·타입·필수 여부가 명세 스키마와 일치한다.
- 에러 응답은 공통 envelope(`{ "error": { "code", "message", "details" } }`) 구조를 따른다(`conventions/api.md`).
- 상태 코드와 `code`가 함께 맞는지 본다(예: 없는 Diary 조회 → `404` + `DIARY_NOT_FOUND`).
- 명세가 코드보다 먼저 옳아야 한다. 불일치는 코드를 고치거나, 근거가 있으면 명세를 먼저 바꾼다(`conventions/api.md`).

## 8. FE 테스트

- MSW 등 목 서버로 화면 단위 동작을 확인한다. 실제 백엔드에 의존하지 않는다.
- 응답 목은 `openapi.yaml`에서 생성된 타입을 기준으로 만들어 명세와 어긋나지 않게 한다.
- 로딩/에러/빈 상태(예: Diary 생성 전 날짜)도 함께 확인한다.

## 9. "완료"의 정의

`conventions/coding.md`와 같다: **테스트 통과 + 명세 일치 + 용어 준수.**
명세 일치는 `harness/`의 hook이 한 번 더 강제한다.
