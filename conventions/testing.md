# 테스트 규칙

이 문서는 **무엇을, 어떻게, 어디까지 테스트할지**에 대한 약속이다.
`conventions/coding.md`(코딩 규칙)·`conventions/api.md`(명세)와 함께 읽는다.
우테코에서 배운 TDD·단위 테스트 원칙을 이 서비스 도메인에 맞춰 정리했다.

## 0. 원칙

- **테스트는 문서다.** 읽는 사람이 "이 코드가 무슨 규칙을 지키는지"를 테스트만 보고 알 수 있어야 한다.
- **가능하면 테스트를 먼저 쓴다(TDD).** 최소한 도메인 로직과 핵심 정책은 테스트로 행동을 고정한 뒤 구현한다.
- **테스트가 없는 도메인 정책은 "구현 안 된 것"으로 본다.** 06:00 KST 경계처럼 말로만 합의된 규칙일수록 테스트로 못 박는다.

## 1. 테스트 종류

| 종류 | 대상 | 도구 | 비고 |
| --- | --- | --- | --- |
| 단위 테스트 | 도메인 객체·서비스 로직 | JUnit 5 + AssertJ + Mockito | 가장 많이, 가장 빠르게 |
| 통합 테스트 | 컨트롤러~리포지토리, DB 포함 | `@SpringBootTest` + Testcontainers PostgreSQL | H2는 대안 |
| 명세 준수 테스트 | API 응답 ↔ `openapi.yaml` | 위 통합 테스트에 포함 | 4절 참고 |
| FE 화면 테스트 | 화면 단위 동작 | MSW 등 목 서버 기준 | TanStack Query 캐시 흐름 포함 |

피라미드를 지킨다: **단위 다수 → 통합 소수**. 모든 것을 무거운 `@SpringBootTest`로 검증하지 않는다.

## 2. 단위 테스트 기준 (BE)

우테코식 단위 테스트의 핵심을 그대로 따른다.

- **given–when–then 구조.** 주석이든 빈 줄이든 세 단계가 눈에 보이게 나눈다.
- **`@DisplayName`은 한국어로 "행위·기대 결과"를 적는다.** 메서드명이 아니라 규칙을 쓴다.
  - 예: `@DisplayName("역질문이 10회에 도달하면 세션은 MAX_FOLLOWUP으로 종료된다")`
- **하나의 테스트는 하나의 개념만 검증한다.** 단언이 여러 규칙에 걸치면 테스트를 쪼갠다.
- **단언은 AssertJ(`assertThat`)로 통일한다.** JUnit `assertEquals`를 섞지 않는다. 예외는 `assertThatThrownBy`.
- **경계값과 예외 경로를 반드시 덮는다.** 정상 케이스만 쓰지 않는다.
  - 역질문 9회/10회/11회, 일자 경계 직전·직후(05:59 / 06:00 / 06:01 KST), 빈 입력, 권한 없는 사용자 등.
- **테스트는 서로 격리한다.** 공유 가변 상태·테스트 실행 순서에 의존하지 않는다.
- **무작위·현재 시각에 의존하지 않는다.** `Clock`(또는 KST 유틸)을 주입해 시간을 고정한다. `LocalDate.now()` 직접 호출 금지.
- **Mock은 외부 의존에만.** 실록이 LLM 호출(`silok/`)·외부 API는 목으로 막고, 도메인 객체 자체는 실제 객체로 검증한다. 도메인 로직을 목으로 가짜 통과시키지 않는다.
- 테스트 클래스명은 `대상클래스 + Test`(예: `DailyChatSessionTest`).

## 3. 이 서비스에서 반드시 테스트할 정책

기획 리뷰에서 합의된 핵심 규칙들이다. 이건 "있으면 좋은" 테스트가 아니라 **필수**다.

- **일자 경계 06:00 KST** — 05:59에 남긴 메모는 전날, 06:00에 남긴 메모는 당일로 묶인다.
- **역질문 최대 10회** — 10회 도달 시 더 질문하지 않고 `endedReason=MAX_FOLLOWUP`으로 종료된다. 서버 정책 레벨에서 막히는지 확인(프롬프트 신뢰 금지).
- **세션 종료 사유** — `USER_ENDED`/`MAX_FOLLOWUP`/`AI_DECIDED`가 올바른 상황에서 기록된다.
- **Diary는 하루 1개** — 하루에 여러 Memo·여러 DailyChatSession이 있어도 06:00 KST 정리 결과 Diary는 날짜당 정확히 하나다.
- **Memo 상태 전이** — `DRAFT → SUMMARIZED`(요약 시), 06:00 KST 이후 `ARCHIVED`. 정의되지 않은 전이는 거부된다.
- **Persona 부분 입력 생성** — 블로그/원문/설문 중 일부만 있어도 Persona가 생성된다.
- **MemoSummary 불변** — MVP에서 수정/삭제 경로가 없다(엔드포인트 부재 또는 거부).

## 4. 명세 준수 테스트

각 도메인은 "응답이 `api/openapi.yaml`과 일치하는가"를 검증하는 테스트를 가진다.

- 성공 응답의 필드명·타입·필수 여부가 명세 스키마와 일치한다.
- 에러 응답은 공통 envelope(`{ "error": { "code", "message", "details" } }`) 구조를 따른다(`conventions/api.md` 4절).
- 상태 코드와 `code`가 함께 맞는지 본다(예: 없는 Diary 조회 → `404` + `DIARY_NOT_FOUND`).
- 명세가 코드보다 먼저 옳아야 한다. 불일치는 코드를 고치거나, 근거가 있으면 명세를 먼저 바꾼다(`conventions/api.md` 8절).

## 5. 통합 테스트 (BE)

- DB가 필요한 검증은 **Testcontainers PostgreSQL**로 한다. 운영 DB와 방언을 맞추기 위함이다(H2는 제약이 있을 때만 대안).
- 06:00 KST 자동 정리 같은 스케줄링 로직은 시각을 주입 가능한 형태로 만들어, 배치 한 번의 결과(전날 데이터 → Diary 1개, 작업 공간 초기화)를 검증한다.
- 보관 정책: 초기화는 **삭제가 아니라 보관**이므로, 초기화 후에도 이전 데이터가 조회·Diary 생성에 쓰이는지 확인한다.

## 6. FE 테스트

- MSW 등 목 서버로 화면 단위 동작을 확인한다. 실제 백엔드에 의존하지 않는다.
- 응답 목은 `openapi.yaml`에서 생성된 타입을 기준으로 만들어 명세와 어긋나지 않게 한다.
- 로딩/에러/빈 상태(예: Diary 생성 전 날짜)도 함께 확인한다.

## 7. 커버리지에 대한 태도

- 커버리지 숫자 자체를 목표로 삼지 않는다. **도메인 로직·핵심 정책(3절)** 을 먼저 덮는다.
- getter나 단순 위임처럼 로직 없는 코드를 위해 억지 테스트를 만들지 않는다.
- 커버리지 임계치를 CI에 강제할지는 `[확정 필요]`(팀 합의).

## 8. "완료"의 정의

`conventions/coding.md` 7절과 같다: **테스트 통과 + 명세 일치 + 용어 준수.**
명세 일치는 `harness/`의 hook이 한 번 더 강제한다.
