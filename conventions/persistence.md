# 저장·트랜잭션 규칙

이 문서는 **무엇을 함께 저장하고, 어디까지 묶고, 실패하면 어떻게 안내할지**에 대한 약속이다.
`conventions/coding.md`(설계)·`conventions/api.md`(에러 응답)·`conventions/testing.md`(실제 DB 검증)와 함께 읽는다.
우테코 레벨2(웹·트랜잭션)에서 정리한 원칙을 이 서비스(회고 도구) 도메인에 맞춰 옮긴 것이다.

## 1. 도메인 ↔ 저장소 분리

- **도메인은 DB를 직접 알지 않는다.** Repository 인터페이스로 통신을 감싼다. 도메인 객체에 SQL·JPA가 새어 나오지 않게 한다.
- **Repository는 Service 계층 안에서만 다룬다.** 컨트롤러·도메인 객체·뷰는 Repository를 모른다.
- 영속성 기술(JPA, 쿼리 방식)이 바뀌어도 도메인 규칙이 흔들리지 않아야 한다.

## 2. 작업 단위(Unit of Work)의 경계 — 무엇을 하나로 묶나

판단 기준은 단 하나: **"일부만 성공한 중간 상태가 사용자에게 거짓을 보여주는가."**

- **묶는다 (하나의 트랜잭션, `REQUIRED`)** — 한 요청 안에서 둘 이상의 데이터를 바꾸고, 그것들이 **함께 성립해야만** 사용자에게 모순 없는 상태가 될 때.
- **묶지 않는다** — 변경이 하나뿐이거나, 여러 변경이라도 서로의 정합성에 영향을 주지 않는 **부수 효과**(알림·로그·통계·캐시 갱신)일 때. 부수 효과는 트랜잭션 경계 밖으로 빼거나 별도 처리한다. 곁다리(알림 실패)가 본질(기록 저장)을 롤백시키면 안 된다.

**경계를 정하는 순서:**

1. 실패하면 사용자가 보는 데이터가 거짓이 되는가? → **묶는다** (`REQUIRED`)
2. 실패해도 본질이 멀쩡하고 나중에 복구 가능한가? → **뺀다** (경계 밖 / `REQUIRES_NEW`)

### 이 서비스의 작업 단위 예시

- **Memo 요약하기** — `Memo`를 `SUMMARIZED`로 바꾸고 + 카테고리별 `MemoSummary`를 만들고 + 새 빈 `DRAFT` Memo를 만드는 흐름은 **함께 성립**해야 한다. 일부만 되면 사용자가 모순된 작업 공간을 본다 → 하나로 묶는다.
- **06:00 KST 자동 정리** — 전날 Memo/MemoSummary/ChatMessage를 모아 **Diary 1개** 생성 + 이전 Memo `ARCHIVED` + 새 작업 공간 시작. Diary 하루 1개 보장과 직결되므로 정합성 단위로 묶는다. (초기화는 **삭제가 아니라 보관**임에 주의 — `conventions/api.md`.)
- **세션 종료** — `status=ENDED` + `endedReason` 기록은 함께 성립해야 한다.
- **EventLog** — append-only **부수 효과**다. 본질 트랜잭션을 롤백시키지 않도록 경계 밖에서 다룬다.

## 3. 저장 시점

- 시스템 상태가 바뀌는 **최소 유효 단위마다** 영속화한다. 한 작업 단위가 끝나면 그 결과를 미루지 않고 저장해, 장애 시 데이터 유실과 중간 상태를 최소화한다.
- 단, 묶지 않아도 되는 것을 한 트랜잭션에 끌어모아 길게 잡지 않는다(2절).

## 4. 실패했을 때 — 원인 종류로 안내를 가른다

롤백은 데이터만 깨끗이 한다. **"다음에 뭘 해야 하는가"는 알려주지 않는다.** 실패 원인의 종류에 따라 안내를 다르게 한다(상태코드 매핑은 `conventions/api.md`).

- **일시적 실패**(락·타임아웃) → "잠시 후 다시 시도해주세요."
- **도메인 규칙 위반**(역질문 10회 초과 시도, 기간 내 Diary 없음 등) → 이유를 알리고 다른 행동으로 안내한다. 재시도해도 같은 결과다.
- **결과 불확실**(응답 유실) → 재시도를 안내하기 전에 **멱등성**을 먼저 보장한다. 멱등성 없는 재시도는 중복을 만든다.

## 5. 동시성과 멱등성

- **재시도가 항상 옳은 건 아니다.** "조회 후 처리"는 그 자체가 경쟁(race)에 노출된다. 멱등성은 애플리케이션 코드의 사전 조회가 아니라 **DB의 원자적 제약·락 같은 단일 직렬화 지점**에서 보장한다.
  - 예: "하루 Diary 1개"는 코드의 중복 검사보다 **유니크 제약**(사용자+KST 날짜)으로 못 박는 편이 안전하다.
- **외부 부수 효과**(이미 호출된 LLM, 발송된 알림)는 되돌릴 수 없어 멱등성에 한계가 있다. 본질 트랜잭션과 분리하고, 재실행해도 안전하도록 설계한다.

## 6. 테스트

- 트랜잭션·롤백·제약조건처럼 **DB의 동작 자체가 검증 대상**이면 실제 DB(Testcontainers PostgreSQL)로 본다. Fake로 빼면 검증 가치가 사라진다(`conventions/testing.md` 3절).
- DB가 단순 협력자이고 그 위 비즈니스 로직만 보고 싶으면 Fake(인메모리 Repository)를 쓴다.
- 06:00 KST 자동 정리는 시각을 주입 가능하게 만들어, 배치 한 번의 결과(전날 데이터 → Diary 1개, 작업 공간 보관·초기화)를 검증한다.
